ViBe Audio Transcript — Manual

YouTube URL → timestamped transcript → template-driven question bank CSV, generated by a self-hosted LLM.

v1.1.3 last updated 2026-07-21 repo: Dheenas-1525/vibe-audio-transcript

Overview

ViBe Audio Transcript is a small self-hosted web app with two chained stages:

  1. Transcription — paste a YouTube URL, get a timestamped transcript back in the browser, downloadable as TXT / SRT / VTT. Runs fully local using yt-dlp + faster-whisper, no external API calls.
  2. Question bank generation — upload a question-bank template CSV, and the app generates matching questions from the transcript using a self-hosted LLM (Qwen3, served via vLLM's OpenAI-compatible API), downloadable as a CSV that matches your template's exact columns.

Everything runs in a single Docker container (app.py, FastAPI) with an in-memory job store — no database.

How it works, step by step

Transcription job lifecycle

POST /api/transcribe spawns a background thread (run_job) that moves a job through these statuses, polled every 1.5s by the browser via GET /api/status/{job_id}:

queued → downloading → transcribing → done (or error)

Audio is downloaded to a temp directory (tempfile.mkdtemp()), transcribed, then deleted immediately — the app never keeps the MP3 around, only the resulting segments list (start/end/text per line).

Question bank job lifecycle

Only available once a transcription job reaches done. POST /api/generate-questions/{job_id} (multipart: template CSV file + questions_per_segment) spawns run_qb_job, polled via GET /api/qb-status/{job_id}:

queued → generating → done (or error)

Internally, qb_generator.generate_question_bank():

  1. Splits the transcript's segments into consecutive ~150-second windows (chunk_segments). The last data point in each window becomes that window's label — the end boundary, not the start, matching the convention observed in reference question banks (the timestamp marks the point by which that segment's content has been covered).
  2. For each window, calls the vLLM server asking for N true/false persona-statement questions strictly from that window's text.
  3. Validates each returned item has all required fields and a valid correct_answer (A or B); drops anything malformed.
  4. Assigns a running Segment index and global S.No., and maps each question onto whatever columns the uploaded template CSV defines — the column set is never hardcoded, only the question style is.

Setup & deployment

Local / server run

git clone git@github.com:Dheenas-1525/vibe-audio-transcript.git
cd vibe-audio-transcript
cp .env.example .env   # fill in VLLM_API_BASE / VLLM_MODEL (see Configuration)
docker compose up --build -d

The Whisper model downloads on first use and is cached in a named Docker volume (whisper-models), so subsequent restarts don't re-download it.

Deploying an update

# on your machine
git add -A && git commit -m "..." && git push

# on the server
cd ~/vibe-audio-transcript
git pull
docker compose up --build -d
Always git pull before rebuilding on the server — if you ever edit files directly on the server, commit or stash them first, or the pull will conflict.

Configuration

All configuration is environment variables, set via a .env file next to docker-compose.yml (auto-loaded by docker compose).

VariableDefaultPurpose
WHISPER_MODELsmallfaster-whisper model size: tiny / base / small / medium / large-v3. Bigger = slower, more accurate.
VLLM_API_BASE(required)Base URL of your vLLM OpenAI-compatible server, e.g. http://172.16.13.91:8002/v1. Do not include /chat/completions — the client SDK appends that.
VLLM_API_KEYEMPTYAPI key for the vLLM server. Most self-hosted vLLM deployments don't check it, but the OpenAI SDK requires some non-empty value.
VLLM_MODEL(required)Exact served model ID, e.g. Qwen/Qwen3-30B-A3B — must match what GET <VLLM_API_BASE>/models reports.
WEB_PORT8000Host-side port the app is exposed on (container always listens on 8000 internally). Change this if 8000 is already taken on the server.
The app has no authentication. Anyone who can reach its port can trigger YouTube downloads and LLM calls on your server/model. Keep it behind SSH tunneling or a private network unless an auth layer is added.

Accessing the app

From your own machine, via VS Code Remote-SSH

  1. Open the Ports tab in VS Code (bottom panel).
  2. Click Forward a Port, enter 8000 (or your WEB_PORT).
  3. Click the browser icon next to the forwarded row, or open http://localhost:8000 in your local browser.

From a plain terminal (no VS Code)

ssh -L 8000:localhost:8000 <user>@<server-host>

Leave that session open, then browse http://localhost:8000 on your own machine.

Public access

Not yet configured. The server's public IP / domain (e.g. an existing vibe.sastra.edu) may already serve a different, unrelated production app on ports 80/443 — don't reuse its path space for this tool. A separate subdomain with its own nginx server block reverse-proxying to 127.0.0.1:8000 is the recommended path when public access is needed; see the project's chat history / commit notes for the exact nginx snippet used during setup discussions.

Using the app

  1. Paste a YouTube URL into the input box, click Transcribe.
  2. Status updates live: Downloading audio… → Transcribing… (mm:ss processed) → done.
  3. Once done, the timestamped transcript renders in the page, with TXT/SRT/VTT download links above it.
  4. A second card appears with two inputs — see below — then click Generate Question Bank.
  5. Progress shows as Generating questions… chunk X/Y. When done, a Download Question Bank CSV link appears.

What to upload, and what the number means

FieldWhat it is
Choose file The question-bank template CSV (e.g. QB - template_Sheet1.csv). The app only reads its header row to know which columns to fill — you can upload either the blank template or one of your already-filled example CSVs (like preparation_of_soap_QB.csv); either works, since the data rows below the header are ignored.
Questions per segment default 5 The transcript is auto-split into ~2.5-minute chunks ("segments" — see How it works). This number is how many questions to generate per chunk, not the total. A 10-minute video (≈4 segments) at the default of 5 yields ~20 questions total. Raise it for more questions per topic chunk, lower it for fewer.

Question bank format

The generator reproduces a specific style discovered from an existing hand-made question bank (preparation_of_soap_QB.csv): each question is a true/false persona statement, not a plain multiple-choice question.

Example row (abbreviated)
QuestionA junior chemist describes soap as the sodium or potassium salt of long-chain fatty acids produced by base hydrolysis of fats or oils.
Option ATrue — "Correct. Soap is precisely defined as..."
Option BFalse — "Incorrect. This is the accurate definition..."
Option C / D(always blank)
Correct AnswerA

Column names and order always come from whatever template CSV you upload — the app reads only its header row (Segment, Question Timestamp [mm:ss], S.No., Question, Hint, Option A, Expln-A, ...) and maps generated content onto those exact columns. The true/false style itself is fixed in qb_generator.py's STYLE_EXAMPLE_ROWS few-shot example — change that constant to switch to a different question style (e.g. full 4-option MCQs) for a different course template.

Troubleshooting

Question generation fails or returns garbage

Verified live against a vLLM 0.19.0 server serving Qwen/Qwen3-30B-A3B: this build does not have a guided-decoding backend enabled, so the naive guided_json parameter is silently ignored. The app instead uses response_format: {"type": "json_object"} (which this server does honor) plus chat_template_kwargs: {"enable_thinking": false} to stop Qwen3 from wrapping a <think>...</think> block around its answer. If you point this app at a different vLLM/model setup and generation breaks again, check the same two things first:

  • Does curl <VLLM_API_BASE>/chat/completions with response_format: {"type":"json_object"} return clean JSON?
  • Does the model wrap output in a reasoning/think block by default, and does it support disabling that via chat_template_kwargs?

Can't reach the app in a browser

  • Confirm the container is actually up: docker compose ps (status should be Up, not Restarting).
  • Confirm it responds locally on the server: curl -s -o /dev/null -w "%{http_code}" http://localhost:8000 should print 200.
  • If that works but a remote browser still can't reach it, it's a network/firewall/NAT issue, not an app issue — see Accessing the app.

Transcription is slow

This deployment runs Whisper on CPU only (device="cpu" in app.py — no GPU wired in). That's inherently much slower than realtime for anything past the tiny/small models; expect a 20-minute video to take several minutes to transcribe, more with medium/large-v3. Two things affect this:

  • WHISPER_MODEL — smaller = faster. Check what's actually set: grep WHISPER_MODEL .env.
  • CPU thread usage — get_model() passes cpu_threads=os.cpu_count() so faster-whisper uses every core the container can see, rather than CTranslate2's conservative default.

If a GPU becomes available later, switching device="cpu" → device="cuda" plus adding the NVIDIA container runtime to docker-compose.yml is typically a 5-10x speedup.

Port 8000 already in use

Set WEB_PORT in .env to something else, e.g. WEB_PORT=8080, then docker compose up -d. The container's internal port never changes — only the host-side mapping.

Known limitations

  • No authentication. Anyone reaching the port can use it.
  • Audio is not persisted. The downloaded MP3 is deleted right after transcription — there is no way to retrieve it after the fact.
  • In-memory job store. The jobs dict lives in process memory only — restarting the container loses all job history and generated question banks that haven't been downloaded yet.
  • Sequential chunk generation. Question-bank chunks are generated one at a time (not parallelized) to avoid overloading the shared vLLM server — long videos with many chunks take proportionally longer.
  • Single question style baked in. The true/false persona format is hardcoded as the few-shot example; a genuinely different question style needs a code change to STYLE_EXAMPLE_ROWS, not just a different template upload.
