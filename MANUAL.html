ViBe Audio Transcript — Manual

YouTube URL → timestamped transcript → template-driven question bank CSV, generated by a self-hosted LLM.

v1.7.0 last updated 2026-07-21 repo: Dheenas-1525/vibe-audio-transcript

Overview

ViBe Audio Transcript is a small self-hosted web app with two chained stages:

  1. Transcription — paste a YouTube URL, get a timestamped transcript back in the browser, downloadable as TXT / SRT / VTT. Runs fully local using yt-dlp + faster-whisper, no external API calls.
  2. Question bank generation — upload a question-bank template CSV, and the app generates matching questions from the transcript using a self-hosted LLM (Qwen3, served via vLLM's OpenAI-compatible API), downloadable as a CSV that matches your template's exact columns.

Everything runs in a single Docker container (app.py, FastAPI), with job state cached in memory and persisted to a small SQLite file so it survives restarts (see Configuration).

How it works, step by step

Transcription job lifecycle

POST /api/transcribe spawns a background thread (run_job) that moves a job through these statuses, polled every 1.5s by the browser via GET /api/status/{job_id}:

queued → downloading → transcribing → done (or error)

Audio is downloaded to a temp directory (tempfile.mkdtemp()), transcribed, then deleted immediately — the app never keeps the MP3 around, only the resulting segments list (start/end/text per line).

Question bank job lifecycle

Only available once a transcription job reaches done. POST /api/generate-questions/{job_id} (multipart: template CSV file + questions_per_segment) spawns run_qb_job, polled via GET /api/qb-status/{job_id}:

queued → generating → done (or error)

Internally, qb_generator.generate_question_bank() runs two phases:

  1. Segment planning (plan_segment_boundaries) — one vLLM call asking the model to propose topic/"teachable beat" boundaries (target 2:30, max 3:00), each snapped to a real transcript sentence-end. Code then enforces the numeric rules regardless of what the model actually returns: _merge_short_segments collapses anything under 1:30 (guards against the model proposing a boundary at every sentence), and _enforce_hard_ceiling force-splits anything still over 3:30 at a real sentence break. Videos shorter than 3:00 skip this call entirely and become a single segment. Each segment's timestamp is its end boundary, not start, matching the convention observed in reference question banks (the timestamp marks the point by which that segment's content has been covered).
  2. Per-segment generation — for each finalized segment, one vLLM call asking for N scenario-based true/false questions strictly from that segment's text, following the detailed style rules in Question bank format below (word counts, naming ratio, curly-quote convention, etc.).
  3. Validates each returned item has all required fields and a valid correct_answer (A or B); drops anything malformed.
  4. Assigns a running Segment index and global S.No., and maps each question onto whatever columns the uploaded template CSV defines — the column set is never hardcoded, only the question style is.

Setup & deployment

Local / server run

git clone git@github.com:Dheenas-1525/vibe-audio-transcript.git
cd vibe-audio-transcript
cp .env.example .env   # fill in VLLM_API_BASE / VLLM_MODEL (see Configuration)
docker compose up --build -d

The Whisper model downloads on first use and is cached in a named Docker volume (whisper-models), so subsequent restarts don't re-download it.

GPU hosts (e.g. a DGX node)

docker-compose.yml stays GPU-agnostic on purpose, since it's shared with CPU-only deployments (referencing an nvidia device reservation on a host without the NVIDIA container toolkit can fail to start the container even at zero GPUs requested). docker-compose.gpu.yml holds the GPU device reservation as an opt-in overlay.

On a GPU host, make plain docker compose up --build -d apply it automatically — no -f flags to remember — by copying it into Compose's auto-loaded override file (gitignored, so this stays local to this machine and never gets committed/pulled to other deployments): 

cp docker-compose.gpu.yml docker-compose.override.yml
docker compose up --build -d

(Or, without creating that file, run docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build -d explicitly every time instead.)

And in .env on that host:

WHISPER_DEVICE=cuda
WHISPER_COMPUTE_TYPE=float16

The Dockerfile's base image is nvidia/cuda:12.4.1-cudnn-runtime-ubuntu22.04 for both CPU and GPU deployments — one shared image instead of maintaining two Dockerfiles. With WHISPER_DEVICE=cpu it behaves exactly like the old python:3.12-slim base functionally, just ~2GB larger on disk; that tradeoff was accepted for simplicity rather than optimizing image size per deployment.

Deploying an update

# on your machine
git add -A && git commit -m "..." && git push

# on the server
cd ~/vibe-audio-transcript
git pull
docker compose up --build -d
Always git pull before rebuilding on the server — if you ever edit files directly on the server, commit or stash them first, or the pull will conflict.

Accessing the app

From your own machine, via VS Code Remote-SSH

  1. Open the Ports tab in VS Code (bottom panel).
  2. Click Forward a Port, enter 8000 (or your WEB_PORT).
  3. Click the browser icon next to the forwarded row, or open http://localhost:8000 in your local browser.

From a plain terminal (no VS Code)

ssh -L 8000:localhost:8000 <user>@<server-host>

Leave that session open, then browse http://localhost:8000 on your own machine.

Public access

Not yet configured. The server's public IP / domain (e.g. an existing vibe.sastra.edu) may already serve a different, unrelated production app on ports 80/443 — don't reuse its path space for this tool. A separate subdomain with its own nginx server block reverse-proxying to 127.0.0.1:8000 is the recommended path when public access is needed; see the project's chat history / commit notes for the exact nginx snippet used during setup discussions.

Using the app

  1. Paste a YouTube URL into the input box, click Transcribe.
  2. Status updates live: Downloading audio… → Transcribing… (mm:ss processed) → done.
  3. Once done, the timestamped transcript renders in the page, with TXT/SRT/VTT download links above it.
  4. A second card appears with two inputs — see below — then click Generate Question Bank.
  5. Progress shows as Generating questions… chunk X/Y. When done, a Download Question Bank CSV link appears.

What to upload, and what the number means

FieldWhat it is
Choose file The question-bank template CSV (e.g. QB - template_Sheet1.csv). The app only reads its header row to know which columns to fill — you can upload either the blank template or one of your already-filled example CSVs (like preparation_of_soap_QB.csv); either works, since the data rows below the header are ignored.
Questions per segment default 5 The transcript is auto-split into ~2.5-minute chunks ("segments" — see How it works). This number is how many questions to generate per chunk, not the total. A 10-minute video (≈4 segments) at the default of 5 yields ~20 questions total. Raise it for more questions per topic chunk, lower it for fewer.

Question bank format

Each question is a scenario-based true/false statement: a fictional decision-maker (founder, developer, team lead, junior/senior engineer, etc.) applies the segment's idea correctly or misapplies it, and the reader judges which. This is a full, detailed spec (not just a style example) baked into qb_generator.py's _segment_prompt() — the rules below are authoritative and take precedence over the looser STYLE_EXAMPLE_ROWS few-shot example (kept mainly for tone/register grounding).

Segmentation

  • Topic-based ("teachable beat"), not fixed time windows — see How it works.
  • Target 2:30 per segment, max 3:00, hard ceiling 3:30 (code-enforced).
  • Every boundary is chosen to land on terminal punctuation (a real sentence-end — a transcript line ending in . ! ?, not just a Whisper pause boundary) whenever one exists within a 20-second tolerance of the target point; _is_terminal_punctuation() checks this against the actual transcript text, and build_summary() flags any final anchor that still doesn't land on one. The last boundary always equals the video's final timestamp.

Per segment (N questions, default 5)

  • 100% true/false: Option A is always "True", Option B always "False", C/D always blank.
  • Question 1 is a straightforward recall check; the rest require application/analysis.
  • Roughly half factual, half a small misconception-surfacing twist.
  • Naming is rare and positional, not left to chance: exactly one question per batch of 5 (~20%) may name a persona, and which question number gets the name rotates across segments — the other questions must use a plain unnamed role ("a developer," not "Arjun, a developer"). Preferred names: Priya, Arjun, Meera, Karthik, Anjali, Ravi, Divya, Vikram, Nisha, Suresh.
  • Question length: 15-30 words median, 38-word ceiling for prose; a fenced code block (3-6 lines) is allowed for code-related content.
  • No meta-references in the question ("the lecture," "the video," "the speaker") — the scenario must stand alone.
  • No closing phrase that signals the answer — present the scenario and let the reader judge.
  • Correct-answer balance is tracked across the whole file, not just per segment: each segment's prompt is told the running True/False ratio so far and nudged to correct any skew, targeting an overall 45-55% split.

Hints

5-15 words, phrased as a pointed question that signals what to check — never gives away the answer.

Explanations (Expln-A / Expln-B)

  • Both are full teaching moments, 40-90 words each, regardless of which option is correct.
  • Start with exactly "Correct." or "Incorrect." (with the period).
  • Include one short direct quote from the transcript (5-25 words) in curly typographic quotes — ‘like this’, not straight quotes — while keeping ordinary apostrophes straight for contractions/possessives (don't, Priya's). Primarily prompt-driven; _wrap_curly_quotes() is a regex safety net that converts a straight 'paired phrase' to curly quotes when the model doesn't, matched narrowly (3-200 chars, no embedded apostrophe) so a lone contraction/possessive apostrophe never matches — it has no closing partner.
  • Ends with a sharp one-line summary where it fits.

Self-check: three tiers

The content spec's self-check list is split into three enforcement tiers, not treated as one flat list:

  • Hard, retry-triggering (_validate_questions): wrong question count for the segment, an explanation not starting with "Correct."/"Incorrect.", or a question containing a banned meta-phrase ("the lecture," "the video"). Any of these makes the whole segment's generation call retry (shared retry budget with network/parse failures).
  • Soft, best-effort during generation (_soft_violation_count): explanation word count (40-90) and closing signal-phrases in questions, scored across the same retry attempts as the hard tier — the best-scoring attempt is kept even if not perfect, rather than failing the whole segment over a single imperfect field.
  • Targeted fix-up after generation (rewrite_explanation / fix_up_explanations): any Expln-A/Expln-B still outside 40-90 words after the batch generation call gets one narrow follow-up call asking the model to rewrite just that field to hit the word count while preserving meaning, tone, and its Correct./Incorrect. opening. A single, narrow ask is far more reliable than getting word count right alongside a dozen other simultaneous constraints in the original generation call. It keeps the closest-to-range attempt across its own retry budget (never reverts to the original, known-worse text just because a rewrite attempt wasn't perfect), and only truly gives up — leaving the original flagged in the summary — if every rewrite attempt fails to even parse.

build_summary() runs after the fix-up pass and re-checks everything (all three tiers, plus timestamp/segment-count/A-B-balance/sentence-break checks that don't fit inside a single generation call) against the final shipped rows. It returns a report shown in the UI below the download link: video length, segment count, question count, A/B split, longest segment, how many segments were code-force-split for exceeding the hard ceiling, how many explanations the fix-up pass rewrote, a sample row, and the specific list of anything still flagged. This is visibility on the final output, not a promise that zero issues ever ship — you decide whether a flagged item is worth editing by hand. Small files (e.g. a single 5-question segment) can trip the A/B-balance check by construction, since 5 discrete items can only land on 0/20/40/60/80/100% — that's an expected limitation of the target on short videos, not a bug.

Example row (abbreviated, live output)
QuestionPriya, a product developer, claims that soap is made via acid hydrolysis of triglycerides.
HintIs acid hydrolysis the correct method for making soap?
Option ATrue — "Incorrect. Saponification is base hydrolysis, not acid hydrolysis. ‘The soap molecule has a long hydrocarbon chain...’ — this implies a base was used..."
Option BFalse — "Correct. Priya has it wrong. Saponification uses a base such as sodium hydroxide, not an acid..."
Correct AnswerB

Column names and order always come from whatever template CSV you upload — the app reads only its header row (Segment, Question Timestamp [mm:ss], S.No., Question, Hint, Option A, Expln-A, ...) and maps generated content onto those exact columns. The content style above is fixed in qb_generator.py's _segment_prompt() — edit that function to change the style for a different course template.

Troubleshooting

Question generation fails or returns garbage

Verified live against a vLLM 0.19.0 server serving Qwen/Qwen3-30B-A3B: this build does not have a guided-decoding backend enabled, so the naive guided_json parameter is silently ignored. The app instead uses response_format: {"type": "json_object"} (which this server does honor) plus chat_template_kwargs: {"enable_thinking": false} to stop Qwen3 from wrapping a <think>...</think> block around its answer. If you point this app at a different vLLM/model setup and generation breaks again, check the same two things first:

  • Does curl <VLLM_API_BASE>/chat/completions with response_format: {"type":"json_object"} return clean JSON?
  • Does the model wrap output in a reasoning/think block by default, and does it support disabling that via chat_template_kwargs?

Can't reach the app in a browser

  • Confirm the container is actually up: docker compose ps (status should be Up, not Restarting).
  • Confirm it responds locally on the server: curl -s -o /dev/null -w "%{http_code}" http://localhost:8000 should print 200.
  • If that works but a remote browser still can't reach it, it's a network/firewall/NAT issue, not an app issue — see Accessing the app.

"HTTP Error 403: Forbidden" when transcribing

This is yt-dlp being blocked by YouTube, not an app bug. Contributing factors: no JS runtime available for yt-dlp to decrypt YouTube's current signature scheme (as of v1.3.2 the image installs deno for this — yt-dlp uses it automatically once it's on PATH, no config needed), an outdated yt-dlp version (YouTube changes frequently; rebuilding pulls the latest via pip install --upgrade), or the server's outbound IP being a datacenter range YouTube rate-limits more aggressively than a residential/campus IP. If 403s persist after rebuilding with the latest image, it's likely IP-based — no code fix for that short of a proxy or cookies-based auth, which isn't currently wired in.

Transcription is slow

This deployment runs Whisper on CPU only (device="cpu" in app.py — no GPU wired in). That's inherently much slower than realtime for anything past the tiny/small models; expect a 20-minute video to take several minutes to transcribe, more with medium/large-v3. Two things affect this:

  • WHISPER_MODEL — smaller = faster. Check what's actually set: grep WHISPER_MODEL .env.
  • CPU thread usage — get_model() passes cpu_threads=os.cpu_count() so faster-whisper uses every core the container can see, rather than CTranslate2's conservative default.

If a GPU becomes available later, switching device="cpu" → device="cuda" plus adding the NVIDIA container runtime to docker-compose.yml is typically a 5-10x speedup.

"Starting..." hangs, or /api/qb-status/<id> / /api/status/<id> returns 404

Before v1.2.0, job state lived only in memory — any container restart (including a routine docker compose up --build -d) silently wiped every job, while a browser tab left open from before the restart still showed the old transcript and kept polling a job_id the server no longer recognized, producing a 404 with no clear explanation. As of v1.2.0 job state is persisted to SQLite, so a plain restart no longer loses finished work. If you still see this: hard-refresh the page (Ctrl+Shift+R) and re-transcribe — the specific job in question was almost certainly still in progress at the moment of a restart, which can't be resumed (see Known limitations).

"Access control checks" / CORS error in the console on generate-questions

If the browser console shows something like Fetch API cannot load http://localhost:8000/api/generate-questions/... due to access control checks even though the address bar shows the exact same origin — you're very likely viewing the page through VS Code's built-in "Simple Browser" preview panel rather than a real, standalone browser window. That panel renders pages inside a restricted internal context that browsers treat as a different origin for CORS purposes, regardless of what URL its address bar displays. Open http://localhost:8000 in your actual browser app instead (a separate window with its own dock/taskbar icon, not a tab inside the VS Code window). As of v1.2.1 the backend also allows CORS from any localhost/127.0.0.1 origin as a defensive measure, in case the real cause turns out to be a port/host mismatch instead.

"CUDA driver version is insufficient" or GPU not visible on a GPU host

This almost always means the container was started without the GPU overlay, not an actual driver mismatch. Confirm with: 

docker compose exec vibe nvidia-smi

If that fails with executable file not found, the GPU was never passed through to the container (the NVIDIA Container Toolkit injects nvidia-smi and the driver libraries into the container only when a device reservation is present). Recreate it with the GPU overlay applied — see GPU hosts above — then re-check with the same command; it should list the host's GPUs.

Port 8000 already in use

Set WEB_PORT in .env to something else, e.g. WEB_PORT=8080, then docker compose up -d. The container's internal port never changes — only the host-side mapping.

Known limitations

  • No authentication. Anyone reaching the port can use it.
  • Audio is not persisted. The downloaded MP3 is deleted right after transcription — there is no way to retrieve it after the fact.
  • Restart recovery is "mark as failed," not "resume." Job state is persisted to SQLite (JOBS_DB_PATH), so completed transcripts and question banks survive a container restart. But a job that was actively downloading/transcribing/generating when the restart happened can't be resumed (its background thread is gone) — it's marked error on the next startup with a message telling you to retry, rather than hanging forever.
  • Sequential chunk generation. Question-bank chunks are generated one at a time (not parallelized) to avoid overloading the shared vLLM server — long videos with many chunks take proportionally longer.
  • Single question style baked in. The true/false persona format is hardcoded as the few-shot example; a genuinely different question style needs a code change to STYLE_EXAMPLE_ROWS, not just a different template upload.
